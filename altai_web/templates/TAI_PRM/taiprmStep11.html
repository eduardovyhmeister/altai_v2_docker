Treat, Terminate, Tolerate and Transfer

Folllowing figure shows the flowchart associated to the 4T's of risk management. The first decision node points to the user to identify if a new failure mode needs to be included. When the user must execute the action, is driven to a decision node named will separate those new conditions and send them through checking alternatives for the 4T's.
When the user do not identify new failure modes, then the decision node "modified KPI risk apetite?"" checks if there were modifications on the risk appetite. If positive, additional KPIs are considered for the risk management process. If negative, the user must evaluate "improved high e-risk?". In this case, the question guides the user to understand if e-risks have been evaluated in a robust mode generating quality enough to finalise the process. If no further system modifications or iteration on KPI analyses is identified, the process ends. Otherwise, before to execute the activity Risk Apetite, previous historical records must be analysed to understand the improvement. The activity Risk Apetite checks if modifications are needed on the risk apetite based on policies established for the risk management process. This means to set up further and stringent conditions for the risks under consideration or to evaluate alternatives to manage the identified risks. This activity, once finished, should start the 4T’s analysis.

Risk Appetite

The risk appetite imposes the corrective actions for the risks (i.e. treat, transfer, terminate or tolerate) in the function of the risk scores and the desire of the user to deal with their risk components. Therefore, these specifications should be defined in the risk policies and easily translated to functional risk matrixes.
